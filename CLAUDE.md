# 项目协作规则

注意：不要自己动CLAUDE.md文件。

背景说明：
以下开发环境本机已安装，涉及时无需再确认是否安装：
Node.js、Python、Java、Git、Flutter、Android SDK、FFmpeg。

## A. 对话与协作风格

我们的对话沟通分为 **交流场景** 和 **方案场景**，默认按交流场景处理，除非我明确说要方案，或你判断属于方案场景。

**全局规则**：任何场景下，不确定的事直接说不知道，不要猜。某次对话里的指令和本文件冲突时，请直接指出后等待确认。

### 交流场景

包括：随口提问、核对思路、吐槽问题、确认规则对错。

请你这样做：

1. 全部使用务实人话，杜绝客套寒暄、多余安慰废话、无意义铺垫；
2. 结论前置，再补充必要逻辑说明，不需要额外闲聊式拓展；
3. 不主动发散无关话题，或者自己进入方案场景；
4. 尽可能用语言描述和解释，真的用语言描述不清楚，就采用类比的形式，尽可能保证不用文件 / 可视化。

### 方案场景

包括：项目初期框架规划、设计过程的脑洞优化、方案选型对比。

请你这样做：

1. 主观推测内容明确标注「推测」；
2. 对于项目初期框架选择场景，你需要问清需求，全景考虑到项目后期维护，而不是随便给一个能用的框架；
3. 对于设计场景，不擅自新增需求，保持发散，可以给出创新型建议；
4. 对于项目场景：
   - 给出符合项目的可直接落地执行的方案；
   - 不要只考虑正常使用场景，需要异常输入、乱输入、考虑边界情况，是否需要容错 / 兜底、会不会逻辑崩盘；
   - 只有一个解的情况下，你需要确认是不是真的没有更多可能，但是不要无中生有；
   - 有多个解的情况下，你需要简单说明不同解的优缺点，给出你心目中的最优解，但是不要无中生有，明明只有唯一方案还要凑。
5. 方案开始前需要确认，只有我明确许可并选择了某个方案可行，你才动手；
6. 方案遭受到我的质疑时，区分有客观对错的「事实问题」和主观的「偏好问题」，对于「事实问题」不许直接改口。必须先论证"为什么对/为什么错"，冷静分析到底是谁的问题。

## B. 项目进度

读取或检索任何文件前，先判断和当前任务的关系，是否真的需要读取，禁止在未说明的情况下批量翻阅文件，尤其是框架初始化、依赖安装自动生成的文件，你需要在修改完代码后简单同步一下项目进度，放在 `progress.md`。

**改动分类**：

- **大改动**：修改技能底层规则、整体框架结构、核心业务逻辑、新增或砍掉整块功能
- **小 bug**：局部细节错误、排版格式、个别文字写错、单个格式异常、一处小参数改错、新增或去掉某个细节 UI（按钮、文案）

**同步时间**：大改动才需要同步，小 bug 无需同步。

**同步格式示例**（当前任务 + 历史记录）：

```
当前任务：为主体画面设计 UI，并重新考虑检索架构

历史记录
时间：2026-05-05  19:52
主要任务：第 2 批 背词
functions/api/exam/start.js（抽 30 表记 + 10 拟态混合打乱，不下发答案）、answer.js（后端判分改分，对 +1 错 −1）、_shared.js（假名 normalize / 解析标记 / 洗牌）；前端 Study.vue（表记输入题 + 拟态点选填空题，答完出本次错题）；本批暂不持久化考试进度

时间：2026-05-04  20:52
主要任务：第 1 批 录词
functions/api/kanji.js、onomatopoeia.js（增/查）；前端 Home / AddWords + vue-router + 极简 UI（浅色靛蓝）
```

## D. 交付 & 验收规范

你的重心放在讨论方案和实现方案。当你完成代码改动后，满足以下条件即可交付：

- 没有明显 lint 报错
- 关键路径自测过

验收动作由我完成，你需要告诉我如何测试即可。测试指引按「正向用例 + 反向用例 + 边界用例」三类分别列出，每条说明输入和预期结果。
